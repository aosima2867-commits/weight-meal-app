# 体重・食事管理アプリ 注意事項 / セキュリティ方針

本ドキュメントは、ユーザーから明示的に指示された開発・運用ルールをまとめたものです。
本アプリの開発・改修・運用に関わるすべての作業は、以下の事項を遵守しなければなりません。

作成日: 2026-04-24
対象アプリ: 体重・食事管理 PWA（プロトタイプ）

---

## 1. Google Drive へのアクセス制限

- アクセスを許可するフォルダは **以下の2つのみ** とする。
  - `食事画像`
  - `健康管理`
- 上記以外のフォルダ・ファイルには **一切アクセスしない**（読み書きとも禁止）。
- 実装上の担保:
  - OAuth スコープは最小権限の `https://www.googleapis.com/auth/drive.file` を使用する
    （ユーザーがアプリで作成・開いたファイルにしかアクセスできないスコープ）
  - 対象フォルダの `folderId` を `.env` に定義し、ファイル操作前に **必ず parent フォルダID を照合** してから読み書きする
  - 照合に失敗した場合は処理を中断し、エラーを表示する

## 2. 個人情報・アカウント情報の取り扱い

- 個人情報、アカウント情報、Google アカウント、メールアドレス、認証トークン等を **一切外部に送信しない**。
- 第三者サーバーへのログ送信、アナリティクス送信、クラッシュレポート送信も行わない。
- 体重・食事データはユーザー自身の Google Drive 上にのみ保存し、開発者やアプリ運用者は参照しない。

## 3. API キー・パスワードの管理

- API キー、パスワード、クライアントシークレット等の秘密情報を **コードに直接書き込まない**。
- 管理方法:
  - ローカル開発: `.env` ファイル（`.gitignore` 対象）に記載
  - PWA 実行時: ユーザー自身が初回セットアップ画面から入力し、ブラウザの `IndexedDB` / `localStorage` に保存（端末内のみ）
  - 公開サンプルは `.env.example` に **空の変数名のみ** 記載
- 対象:
  - Google OAuth Client ID / Secret
  - Claude API Key
  - Google Drive のフォルダID（秘密情報ではないが .env 管理）

## 4. 作業手順ルール

- すべての作業（ファイル作成・編集・コード変更・外部サービス連携）の前に、
  **必ず計画を提示し、ユーザーの承認を得てから実行する**。
- 承認なしに勝手に作業を開始しない。
- 計画には以下を含めること:
  - 変更・作成するファイル名
  - 変更内容の要約
  - 想定される影響範囲
  - 必要なユーザー側の準備（APIキー等）

## 5. 禁止事項（まとめ）

| 区分 | 禁止されていること |
|---|---|
| Drive | 「食事画像」「健康管理」以外のフォルダへのアクセス |
| 通信 | 個人情報・アカウント・認証情報の外部送信 |
| コード | APIキー・パスワードのハードコーディング |
| プロセス | ユーザー承認なしの作業着手 |
| ログ | 第三者サーバーへのアナリティクス・クラッシュレポート送信 |

## 6. 実装上のチェックリスト（v0.3 時点の実装状況）

- [x] `.gitignore` に `.env` が含まれている
- [x] API キーがソースコード内にハードコードされていない（grep 確認済）
- [x] Google Drive API スコープが `drive.file` に限定されている（`DRIVE_SCOPE` 定数）
- [x] ファイル書き込み前に folderId 照合が実装されている（`verifyParent()` を `appendRowToExcel`/`uploadImageToDrive` で必ず呼出）
- [x] 万一、許可外フォルダにアップロードされた場合の自動削除実装（`uploadImageToDrive` 内）
- [x] 外部への fetch 先が `googleapis.com` / `anthropic.com` / `cdn.jsdelivr.net` のみ
- [x] コンソールログに個人情報を出力していない（catch 句のみ警告ログ）
- [x] 新機能追加時、事前にユーザー承認を得ている（毎回計画を提示）

### 防御の二重化
1. **OAuth スコープ `drive.file`**: ユーザー全Drive閲覧を物理的に不可能にする
2. **書き込み前 parent 照合**: 仮にコードに脆弱性があっても、許可した2フォルダ以外には絶対書き込まれない

## 7. 違反時の対応

上記に反する実装・運用が発見された場合は、
ただちに該当機能を停止し、ユーザーに報告・修正方針の承認を得ること。
